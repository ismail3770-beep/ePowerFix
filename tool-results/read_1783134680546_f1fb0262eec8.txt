   163тЖТ   163тЖТ## 8. Architecture Recommendation тАФ Mini-Service vs Next.js API Routes
   164тЖТ   164тЖТ
   165тЖТ   165тЖТ**RECOMMENDATION: Keep the Express mini-service on port 4000 (current architecture). Do NOT convert to Next.js API routes.**
   166тЖТ   166тЖТ
   167тЖТ   167тЖТRationale:
   168тЖТ   168тЖТ1. **Express API is already 90% built and working** тАФ 27 admin route files, comprehensive endpoints, validation via zod, rate-limiting, security middleware, JWT auth. Converting to Next.js routes would be a massive rewrite.
   169тЖТ   169тЖТ2. **Next.js proxy already solves CORS + cookies** тАФ `next.config.ts` rewrites `/api/:path*` тЖТ `${apiUrl}/api/:path*` (default `http://localhost:4000`). The `lib/api.ts` always uses same-origin fetch with `credentials: 'include'`. This is clean and production-ready.
   170тЖТ   170тЖТ3. **Two ports is fine in dev** тАФ `bun run dev` (Next on :3000) + `bun --filter api dev` (Express on :4000). The proxy is transparent to the frontend.
   171тЖТ   171тЖТ4. **The admin panel "static" issue is NOT an API problem** тАФ it's a **routing + onClick handler problem** on the frontend. The APIs exist and work; the pages just don't call them or don't navigate to them properly.
