# Reference-style storefront refresh

The storefront is rebuilt around a reference-style home page, shared product-card treatment, and visual refinements to the cart drawer, product detail, project kits, and checkout. Existing cart, order, checkout, and route contracts remain the integration points; homepage cards now open the existing drawer and use the authenticated wishlist control. The visual layer uses the current `epf-*` palette, `--primary`, and root font providers, while the admin tree has no diff against `main`. **Watch for:** **confirmed** the new hero carousel ignores reduced-motion preferences and has no way to stop rotation; **confirmed** the blog preview lost its eight-second request timeout, so a stalled API response can leave the home page loading forever; **confirmed** neither stateful change has regression coverage in this diff.

## High-level view

Visual scope stays within `epf-reference-*` classes and existing ePowerFix tokens rather than creating a parallel theme. Cart additions, wishlist state, checkout payloads, and routes remain behind their existing boundaries, and `apps/web/src/app/admin` is unchanged.

**Confirmed:** banner data is now a rotating collection instead of the first static banner. The timer runs for every multi-banner response, but neither it nor the changing content respects reduced-motion preferences or exposes a pause action.

**Confirmed:** the blog card redesign replaces the prior bounded request with a bare `apiFetch` call. A connection that stalls rather than fails leaves React Query pending and the blog skeleton visible indefinitely.

**Confirmed:** no regression coverage accompanies the carousel lifecycle or changed blog failure mode. Those browser behaviors need focused tests rather than relying on the repository-wide lint suite.

## Hero rotation bypasses the storefront’s motion preference

**Confirmed:** `HomeHero` starts a five-second interval whenever `/api/banners` returns more than one item (`apps/web/src/app/page.tsx:254–257`). The reference-card reduced-motion rule only disables CSS transitions (`apps/web/src/app/globals.css:591–598`); it does not suppress the timer or freeze hero content. The dot controls select a slide but cannot stop future changes, so users who opt out of motion continue to receive changing hero copy and imagery. Gate autoplay with `matchMedia("(prefers-reduced-motion: reduce)")` and add a persistent pause mechanism.

## Blog loading can remain pending after a network stall

**Confirmed:** the previous home blog query created an `AbortController` and aborted after eight seconds; the replacement at `apps/web/src/app/page.tsx:442–446` calls `apiFetch` directly. `apiFetch` is an unbounded `fetch` wrapper, and this query has `retry: false`, so a request that never resolves does not enter the existing empty or error state. The page therefore keeps its blog skeleton rather than recovering when the upstream or proxy connection stalls. Restore the scoped timeout or standardize a timeout-capable API helper.

## Regression coverage omits changed home behavior

**Confirmed:** no tests are added in the diff for timer-driven banner state or the altered blog failure behavior. Not tested: reduced-motion behavior, timer cleanup after banner data changes, manual slide selection, and the stalled-blog response path; the refreshed card actions also need a smoke check that cart opening and signed-in or signed-out wishlist behavior still reach their existing boundaries. Add focused tests at those integration points.
