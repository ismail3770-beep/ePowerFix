# ProjectKit identity separation and guest booking integrity

Phase 0A gives sellable ProjectKits their own cart and order identity, preserves legacy kit carts through normalization, and permits guest service bookings without synthetic `User` references. The order endpoint now resolves authoritative prices, decrements stock, creates order lines, and reserves coupon usage in one database transaction. Guest booking records retain submitted contact snapshots while valid account links remain optional, and the admin mapping prefers those snapshots. **Watch for:** **confirmed** online-payment failures or cancellations leave newly committed stock and coupon reservations behind; **confirmed** legacy persisted cart records without `productImage` are now discarded.

## High-level view

The canonical ProjectKit path is **confirmed** to carry `itemType: PROJECT_KIT` and `projectKitId` from web producers through cart-to-order conversion, while portfolio Projects retain `projectId`. The migration gives carts and order lines a dedicated foreign key, converts only unambiguous historical kit references, and the public order endpoint retains the legacy `PROJECT` form.

Guest service booking integrity is **confirmed**: guest rows use a nullable `userId` with a name/email snapshot, invalid synthetic references are nulled before the `SET NULL` foreign key is restored, and admin responses prefer the snapshot. Existing account-backed bookings retain their links.

The order transaction makes authoritative pricing, conditional stock decrements, order creation, and limited coupon reservation one atomic operation. Its **confirmed** boundary ends before the unchanged payment initiation and callback flows, so a successfully committed order can remain reserved when gateway setup fails or the customer cancels.

Cart hydration translates the former `PROJECT + productId` ProjectKit convention on web and mobile. Its **confirmed** runtime filter now treats a missing image as an invalid cart record even though the prior shared-store filter accepted it, turning optional display metadata into a compatibility requirement.

### Gateway setup can strand inventory and coupon capacity

The new transaction in `apps/api/src/routes/orders.ts` decrements product, variant, or ProjectKit stock and creates `CouponUsage`/increments `Coupon.usedCount` before returning an `Order(PENDING)`. Both web checkout paths and `apps/mobile/app/checkout.tsx` create that order first, then call `/api/payments` or `/api/payments/initiate`; their error handling leaves the cart available for retry but does not cancel the first order. The unchanged payment routes only mark successful payments as paid, while failure or cancellation redirects to `/payment/fail`; no stock increment or coupon-release path exists.

This failure mode is **confirmed** by the new reservation writes, the client ordering, and the unchanged callback implementation. A gateway outage, rejected initiation, cancelled hosted checkout, or lost initiation response strands the first pending order's stock and coupon capacity; retrying creates another reservation, and an authenticated coupon user can then be blocked by the first order's `CouponUsage` row. Introduce a pending-payment reservation lifecycle with idempotency or reuse of the existing order, plus an atomic release on initiation failure, cancellation, expiry, and administrative cancellation.

```text
POST /orders
  └─ transaction: stock--, coupon usage++, Order(PENDING)
       └─ client calls payment initiation after commit
            ├─ success → existing callback confirms payment
            └─ failure/cancel → redirect or client error; no release
```

### Cart migration drops records that were previously accepted

`packages/store/src/cart.ts` and `apps/web/src/store/cart-store.ts` now require `productImage` to be a string in their hydration validators. Before Phase 0A, the shared validator required an id, product id, name, numeric price, and quantity but did not require an image; web persistence also did not impose this runtime check. The rendering paths tolerate a missing image with a fallback, so this rejection is unrelated to whether a cart line can be checked out.

The regression is **confirmed** for any persisted cart created without `productImage` or with a null image. Accept the historic shape during normalization and write `productImage: ''` (or preserve a nullable image) rather than filtering the line out.

### Phase 0A behavior has no regression coverage

The Phase 0A diff adds no test files, and the existing `apps/web/src/lib/__tests__/api-handler.test.ts` defines its own stale three-value order-item enum rather than importing the API route schema. **Not tested:** `PROJECT_KIT` acceptance and legacy conversion, incomplete persisted carts, guest-booking migration fixtures with synthetic user ids, concurrent stock/coupon reservations, and failed or cancelled online payment attempts.

This gap is **confirmed**. Add focused unit and integration tests for the cart and payload converters, migration fixtures for both valid and synthetic booking ownership, and order/payment tests that prove either release or idempotent reuse of a pending reservation.
