# Payment reservation lifecycle for online orders

Phase 0A turns authenticated online orders into time-bounded stock and coupon reservations, with conditional release or payment commit. The normal path is materially safer: release is claimed once inside a transaction, callbacks bind provider method/transaction/amount, and web cart clearing waits for an owner-authorized `PAID` read. **Watch for:** **Critical — confirmed** the schema migration and expiry cleanup have no deployment execution path; **High — confirmed** a gateway-confirmed `MANUAL_REVIEW` payment can still expire and release inventory; **High — likely** a gateway timeout after acceptance leaves no durable attempt to reconcile.

## High-level view

**Confirmed:** authoritative stock/coupon reservation and the `ACTIVE`→`RELEASED` claim share a transaction. Definitive initiation failures, authenticated callback failures, expiry, and admin cancellation/deletion therefore cannot double-credit a known-unpaid reservation.

**Critical — confirmed:** expiry is a protected HTTP handler and bounded helper only. Nothing schedules it, and neither deployment path applies the migration that makes its state fields exist.

**High — confirmed:** callback completion binds method, transaction, and authoritative amount correctly, but payment uncertainty is mishandled. `MANUAL_REVIEW` can expire, browser-returned bKash/Nagad definitive failures stay pending, and an unknown initiation result is not stored before the provider call.

**High — likely:** POST callback allowlisting fails closed in production but trusts `req.ip` behind a fixed one-hop proxy setting. Its security depends on ingress preventing direct access and sanitizing forwarding headers.

**Confirmed:** clients retain carts until verified success, but the pending-order pointer is session-only on web and memory-only on mobile. It is also detached from the cart snapshot, so session loss or cart edits can produce a duplicate reservation or erase later cart lines.

1. **[Critical · confirmed] Database migration is not deployed** — configured deploy paths generate Prisma client code but never run `prisma migrate deploy`. Run migrations as a serialized release step before starting the API.
2. **[Critical · confirmed] Expiry cleanup has no invoker** — `/internal/release-expired` and `releaseExpiredOrderReservations` have no scheduler, worker, or deployment cron caller. Schedule the protected endpoint or a durable worker, with retries and monitoring.
3. **[High · confirmed] Manual-review payments can release captured inventory** — provider-success anomalies set `MANUAL_REVIEW`, yet expiry releases every non-`PAID` reservation. Add a payment-uncertain hold and require reconciliation/refund before release.
4. **[High · likely] Unknown initiation outcomes are unreconcilable** — gateway timeouts leave no local attempt even though the provider may have accepted the request. Persist an initiating attempt and merchant correlation before the outbound call, then reconcile before retry.
5. **[High · confirmed] Definitive bKash/Nagad browser failures remain reserved** — GET callbacks validate against the provider but always redirect pending; only POST callbacks release. Release when server validation is definitive and add an authenticated path for returns without a provider ID.
6. **[High · likely] Callback IP authorization depends on a spoofable proxy contract** — `trust proxy = 1` is safe only behind one header-sanitizing proxy with no direct ingress. Trust explicit proxy CIDRs, block direct access, and test spoofed forwarding headers.
7. **[Medium · confirmed] SSLCommerz cancellation capability is logged in the URL** — the signed fail token is a query parameter before production access logging. Redact it at every logging layer and avoid a reusable URL capability where the gateway contract permits.
8. **[Medium · likely] Expiry is not enforced atomically at the final claim** — lock and commit predicates omit `reservationExpiresAt > now`. Put the deadline in those predicates and re-read after locking.
9. **[High · confirmed] Authenticated users can hoard active reservations** — online order creation has no active-reservation cap or order-specific limit, while inventory is decremented before payment. Limit active unpaid orders per account and rate-limit online order creation.
10. **[High · confirmed] Pending payment recovery is not durable across mobile restarts or web session loss** — the cart survives while the order pointer and idempotency key do not. Persist/recover active orders server-side before allowing another order.
11. **[Medium · confirmed] Pending checkout is not cart-snapshot-bound** — web retry reopens the old order while cart controls remain editable, then verified success clears every current line. Freeze the cart or clear only the purchased snapshot.
12. **[High · confirmed] No regression suite covers the reservation boundary** — no changed test files exercise expiry, release races, gateway uncertainty, callback trust, or client recovery. Add API integration and client-state coverage before shipping.

<summary>Details</summary>

## The release claim keeps known-unpaid cleanup idempotent

`releaseOrderReservation` conditionally moves only an active, non-`PAID` order to `RELEASED`, then restores only `inventoryReserved` lines and removes the matching coupon usage (`apps/api/src/lib/order-reservations.ts:63–156`). The claim and restoration share a transaction, so callback, expiry, and admin races cannot restore stock twice. The helper is used for definitive initiation/callback failures and expiry (`apps/api/src/routes/payments.ts:407–410, 492–493, 557–578`) and both admin cancellation paths plus deletion (`apps/api/src/routes/admin/orders.ts:307–310, 379–381, 413–416`).

Creation has the right authority boundary: online checkout requires an account before mutation (`apps/api/src/routes/orders.ts:274–281`), inventory decrements are conditional, and only online physical lines are marked reserved (`apps/api/src/routes/orders.ts:337–453, 568–570`). The remaining release problems are uncertainty and execution, not duplicate restoration for a known-unpaid order.

## Production never applies the reservation schema

**Critical — confirmed:** the migration adds the fields used by `Order` reads (`prisma/migrations/20260718010000_phase_0a_payment_reservations/migration.sql:4–21`), but Railway only runs `prisma generate` before starting (`apps/api/railway.json:3–11`). The Docker path likewise generates a client and starts `tsx src/server.ts` without migration (`Dockerfile:27–40`). An existing database will therefore lack queried columns and fail ordinary order reads and checkout. Run `prisma migrate deploy --schema=prisma/schema.prisma` in a serialized migration job before API replicas start.

## Expiry is implemented but never runs

**Critical — confirmed:** `/internal/release-expired` authorizes a secret and calls the expiry helper (`apps/api/src/routes/payments.ts:422–438`); the helper scans expired active reservations and releases each in a transaction (`apps/api/src/lib/order-reservations.ts:173–207`). Repository-wide search found no scheduler, cron definition, worker loop, CI job, Railway configuration, or caller. No expiry scheduler currently invokes the protected cleanup endpoint or release helper, so abandoned orders hold stock and coupon capacity until some unrelated interaction occurs. Schedule the protected POST or a durable worker at an interval consistent with the 30-minute hold, retry failures, and alert on persistent backlog.

## Manual review follows the unpaid expiry path

**High — confirmed:** provider-success responses with a missing/invalid/mismatched transaction or amount reach `flagPaymentForManualReview` (`apps/api/src/routes/payments.ts:227–300`). It changes payment and order status to `MANUAL_REVIEW` but leaves the reservation active (`apps/api/src/routes/payments.ts:170–201`); expiry and release accept all active statuses except `PAID` (`apps/api/src/lib/order-reservations.ts:71–86, 179–196`). A captured-but-unreconcilable payment can therefore automatically return inventory and coupon capacity. Model a payment-uncertain hold separately, exclude it from automatic/admin release, and require an explicit fulfill/refund decision.

## Gateway initiation is not durable before its external side effect

**High — likely:** the route calls a gateway before creating a usable `Payment` row (`apps/api/src/routes/payments.ts:531–540, 562–641`). SSLCommerz, bKash, and Nagad classify transport failures as `UNKNOWN` (`apps/api/src/lib/payments.ts:348–354, 515–521, 676–680`), while that route branch neither releases nor persists an attempt (`apps/api/src/routes/payments.ts:542–560`). A provider that accepted a timed-out request later cannot be found because callback lookup starts with a stored method/transaction record (`apps/api/src/routes/payments.ts:150–161`); retry can create a second payment. Persist `INITIATING` state and a merchant-controlled reference before the call, then reconcile it before retrying.

## bKash and Nagad discard definitive browser failures

**High — confirmed:** bKash/Nagad GET handlers run server-side validation but redirect pending for every unsuccessful result (`apps/api/src/routes/payments.ts:683–692, 784–793`). Their POST equivalents release on `validation.failureKind === 'DEFINITIVE'` (`apps/api/src/routes/payments.ts:734–738, 835–839`). The query is untrusted, but a definitive provider validation is not. Invoke `markPaymentFailed` from GET only for a definitive result; retain pending for unknown validation and add signed/server-side cancellation handling where no provider ID is returned.

## Expiry is observed before, not enforced by, the final transition

**Medium — likely:** `isReservationExpired` is checked in JavaScript (`apps/api/src/routes/payments.ts:91–95`), but the initiation lock and paid claim omit an unexpired predicate (`apps/api/src/routes/payments.ts:117–126, 304–312`). The deadline can pass between read and update, committing an initiation or payment after expiration. Add `reservationExpiresAt > now` to active-reservation claims, retain a legacy `NONE` branch if required, and re-read/release when that conditional update fails.

## Callback authorization depends on proxy deployment details

**High — likely:** the verifier correctly uses `req.ip` and fails closed with no production allowlist (`apps/api/src/lib/payment-callback-security.ts:22–42`), but its trust source is `app.set('trust proxy', 1)` (`apps/api/src/server.ts:47–49`). If Express is directly reachable or the upstream forwards caller-supplied `X-Forwarded-For`, a caller can appear allowlisted to the POST handlers (`apps/api/src/routes/payments.ts:715–718, 817–820, 889–892`). Trust only known proxy CIDRs, firewall direct access, and test the deployed forwarding chain; make nonempty provider allowlists a deployment preflight.

## SSLCommerz cancellation is bound correctly but exposed as a logged URL capability

The signed token binds gateway, order, transaction, purpose, and expiry, then the handler verifies its order/transaction match (`apps/api/src/lib/payments.ts:48–152`; `apps/api/src/routes/payments.ts:860–885`). The pending-state transition prevents replay from restoring stock twice.

**Medium — confirmed:** that capability is still placed in `fail_url`/`cancel_url` as `?token=...` (`apps/api/src/lib/payments.ts:203–206`) before production `morgan('combined')` logging (`apps/api/src/server.ts:68–70`). It is written to application access logs and may reach proxy/gateway logs; a holder can invoke the unauthenticated fail route before expiry. Redact the query across all logging layers, preserve the existing no-referrer posture, and prefer a single-use server-side capability or provider-authenticated cancellation where feasible.

## Reservations can be hoarded without payment

**High — confirmed:** an authenticated caller can create active online orders without starting payment (`apps/api/src/routes/orders.ts:274–285, 568–570`). There is no active-reservation cap or order-specific limit: the visible API-wide cap is 1,000 requests per IP every 15 minutes (`apps/api/src/server.ts:74–80`), while the five-per-ten-minute rate limit applies only after creation (`apps/api/src/routes/payments.ts:455–459`). Different idempotency keys can hold stock repeatedly, indefinitely while expiry is inert. Cap concurrent active reservations, rate-limit online order creation, and reuse an existing active order for the account/cart.

## Pending checkout state is neither durable nor cart-bound

**High — confirmed:** web stores only the pending order pointer in `sessionStorage` (`apps/web/src/app/checkout/page.tsx:94–128`; `apps/web/src/components/epf/CheckoutDialog.tsx:106–109`) and mobile stores it only in component state (`apps/mobile/app/checkout.tsx:105–107, 247–254`). Persisted cart contents survive while the pointer and cleared idempotency key do not, so restart/session loss can create a second reservation. Recover active orders server-side on checkout/app launch before creating another one.

**Medium — confirmed:** web retries the old order (`apps/web/src/app/checkout/page.tsx:424–439`) while quantity/remove controls remain live (`apps/web/src/app/checkout/page.tsx:852–873`), then clears the entire current cart on verified success (`apps/web/src/app/payment/success/page.tsx:58–62`). Freeze cart mutation or retain an immutable order snapshot and clear only matching purchased quantities.

## The reservation state machine has no regression contract

**High — confirmed:** the diff adds no test file, while CI only invokes repository-wide `bun test` (`.github/workflows/ci.yml:35–49`). There is no coverage for release races, expiry authorization, unknown initiation, callback binding/trust, or restart/cart-snapshot recovery. Add disposable-PostgreSQL API integration tests and focused client-state tests for these failure paths.

## Validation

`bun x prisma validate --schema=prisma/schema.prisma`, web no-emit TypeScript, mobile no-emit TypeScript, and `git diff --check HEAD` passed. Full `bun test --reporter=dots` failed three unchanged cache/rate-limit tests; API-wide no-emit remains red with 144 errors outside this payment-review slice. Neither provides reservation-lifecycle coverage.
